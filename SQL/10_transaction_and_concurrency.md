# 🏦 SQL: Transactions & Concurrency
 
A transaction is a group of SQL statements that represent a **single unit of work**. All statements in a transaction must complete successfully, or none of them will be applied to the database.
 
---
 
## 📑 ACID Properties
 
To ensure data integrity, transactions must adhere to the **ACID** principle:
 
- **Atomicity**: Transactions are like atoms—indivisible. Either all statements succeed, or the entire transaction is rolled back.
- **Consistency**: The database will always transition from one valid state to another. You won't end up with orphaned data (e.g., an order without an item).
- **Isolation**: Transactions are protected from each other. Changes made in one transaction are not visible to others until committed.
- **Durability**: Once a transaction is committed, the changes are permanent and will survive system failures or power outages.
 
---
 
## 🛠️ Managing Transactions
 
By default, most SQL engines (like MySQL) use **Autocommit**, meaning every single statement is treated as a transaction and automatically committed.
 
### Manual Transactions
You can group multiple statements manually using `START TRANSACTION`.
 
```sql
-- Disable autocommit if needed
SET @@autocommit = 0;
SHOW VARIABLES LIKE '%autocommit';
 
START TRANSACTION;
 
INSERT INTO sql_store.orders (customer_id, order_date, status)
VALUES (1, '2024-04-10', 1);
 
-- LAST_INSERT_ID() gets the ID generated by the previous INSERT
INSERT INTO sql_store.order_items (order_id, product_id, quantity, unit_price)
VALUES (LAST_INSERT_ID(), 1, 1, 10.00);
 
-- Finalize the changes
COMMIT;
 
-- OR, if something went wrong, undo everything
ROLLBACK;
```

## 🔒 Concurrency and Locking
 
Concurrency happens when multiple users try to access or modify the same data at the same time. To prevent data corruption, databases use **Locks**.
 
```sql
USE sql_store;
START TRANSACTION;
UPDATE customers
SET points = points + 10
WHERE customer_id = 1;
COMMIT;
```
 
> [!NOTE]
> **Try it out**: Open two separate connection tabs in MySQL Workbench. Start a transaction in Tab A and update a row. Before committing, try to update the same row in Tab B. You will see a "spinner"—Tab B is waiting for the lock to be released by Tab A.

## ⚠️ Concurrency Problems
 
When isolation isn't strong enough, several problems can occur:
 
### 1. Lost Updates
Occurs when two transactions update the same data simultaneously without proper locking. The last commit overwrites the first one.
![Lost Updates](files/Screenshot%20from%202026-03-26%2008-58-32.png)
 
### 2. Dirty Reads
Happens when a transaction reads data that has been modified by another transaction but **not yet committed**.
![Dirty Reads](files/Screenshot%20from%202026-03-26%2009-01-49.png)
 
### 3. Non-repeatable Reads
Occurs when you read the same data twice during a transaction but get different results because another transaction updated it in between.
![Non-repeating Reads](files/Screenshot%20from%202026-03-26%2009-04-49.png)
 
### 4. Phantom Reads
Similar to non-repeatable reads, but involves **new rows** appearing (or disappearing) in a query result because another transaction added or deleted them.
![Phantom Reads](files/Screenshot%20from%202026-03-26%2009-08-18.png)

## 🪜 Transaction Isolation Levels
 
Isolation levels define how much a transaction is protected from the "noise" of other concurrent transactions.
 
![Transaction Isolation Levels](files/Screenshot%20from%202026-03-26%2009-14-33.png)
 
| Isolation Level | Dirty Reads | Non-repeatable Reads | Phantom Reads | Performance |
| :--- | :---: | :---: | :---: | :--- |
| **Read Uncommitted** | ✅ Yes | ✅ Yes | ✅ Yes | 🚀 Fastest |
| **Read Committed** | ❌ No | ✅ Yes | ✅ Yes | 🏎️ Fast |
| **Repeatable Read** (Default) | ❌ No | ❌ No | ✅ Yes | 🚙 Normal |
| **Serializable** | ❌ No | ❌ No | ❌ No | 🐢 Slowest |

## 🔬 Testing Isolation Levels
 
### 1. READ UNCOMMITTED
Allows you to see uncommitted changes from other transactions.
 
```sql
-- Transaction A
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT points FROM customers WHERE customer_id = 1; -- Sees '20' even if not committed
```
 
### 2. READ COMMITTED
You only see changes that have been finalized (committed).
 
```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- Will show old data until the other user runs COMMIT.
```
 
### 3. REPEATABLE READ (MySQL Default)
Ensures that if you read a row once, you will see the same data throughout the entire transaction, even if another user commits a change.
 
```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT points FROM customers WHERE customer_id = 1; -- Initial read
-- ... Another user commits a change to 40 ...
SELECT points FROM customers WHERE customer_id = 1; -- Still shows original value
COMMIT;
```
 
### 4. SERIALIZABLE
The strictest level. It places a lock on the entire range of data you are reading. Other users cannot even **insert** or **update** rows in that range until you finish.
 
```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- Transaction will wait for others to finish before reading to ensure absolute consistency.
```

## 💀 Deadlock
 
A deadlock occurs when two transactions are both waiting for the other to release a lock, causing them to get stuck forever.
 
```sql
-- Transaction A
UPDATE customers SET state = 'VA' WHERE customer_id = 1;
UPDATE orders SET status = 1 WHERE order_id = 1; -- Waiting for Transaction B

-- Transaction B
UPDATE orders SET status = 1 WHERE order_id = 1;
UPDATE customers SET state = 'VA' WHERE customer_id = 1; -- Waiting for Transaction A
```
 
> [!IMPORTANT]
> **Prevention**: To minimize deadlocks, always update tables in the **same order** across all your scripts.
